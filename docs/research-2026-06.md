# State-of-the-art research notes — 2026-06-11

(6 parallel research agents + completeness critic; sources cited inline.)


## oss-landscape

### Rewind.ai (teardown) never used embeddings for search: pure SQLite FTS4 keyword search with Porter stemming, plus rich per-frame metadata
Kevin Chen's teardown: screenshots every 2s, encoded to H.264 at 0.5fps via bundled libx264/FFmpeg (~496 kb/s, ~5-min chunks, ~180 MB/hr video, 26 MB/hr text). OCR via Apple Vision (Live Text pipeline). Schema: `frame` (one row per video frame), `search_content` + FTS virtual table (tokenize=porter), `node` (maps each OCR word to normalized 0-1 frame coordinates), `segment` (app/window changes with browser URL). No encryption at rest. The lesson: a wildly loved product shipped with keyword FTS + app/window/URL metadata, not semantic search.
— https://kevinchen.co/blog/rewind-ai-app-teardown/

### screenpipe (YC S26, Rust) abandoned fixed-cadence capture for event-driven capture and uses the accessibility tree as PRIMARY text source, OCR only as fallback
Captures on OS events (app switch, click, typing pause, scroll, clipboard; ~5s idle fallback), pairing each screenshot with an accessibility-tree walk at the same timestamp. macOS fallback OCR = Apple Vision; Windows = native OCR; Linux = Tesseract. Storage: SQLite (~/.screenpipe/db.sqlite) with tables frames (snapshot_path, accessibility_text, capture_trigger), ocr_text, elements (FTS5), ui_events; JPEGs on disk (~300 MB/8hr). No embeddings at all — search is FTS5 + REST API (localhost:3030, /search, /raw_sql) plus 'pipes' (cron-scheduled .md prompt agents). Claimed footprint: ~600 MB RAM, 5-10% CPU, 5-10 GB/month on M3.
— https://docs.screenpi.pe/architecture

### OpenRecall has the exact same retrieval architecture as ScreenMemory — and the same relevance problem: one vector per whole screen via a small sentence-transformer
openrecall/nlp.py: MODEL_NAME = "all-MiniLM-L6-v2"; get_embedding() splits OCR text into lines, embeds each line, then returns np.mean(sentence_embeddings, axis=0) — a single averaged vector per screenshot, brute-force cosine in NumPy, embeddings stored as blobs in SQLite. OCR is docTR, screenshots are similarity-deduped WebP. This 'average the whole screen' design is widely reported as giving mediocre relevance — independent confirmation that ScreenMemory's pain comes from chunking/granularity, not from the specific model choice.
— https://raw.githubusercontent.com/openrecall/openrecall/main/openrecall/nlp.py

### Microsoft Recall is the only system doing region-level indexing: per-region OCR with bounds, per-word boxes, DiskANN vector indexes, and SQLite encrypted inside a VBS enclave
TotalRecall analysis of ukg.db (SQLite SEE, AES-256-GCM, keys TPM-sealed in a VBS enclave; all queries execute inside the enclave, gated by Windows Hello): 17 tables incl. WindowCapture (WindowTitle, ActivationUri, DwellTime...), ScreenRegion (RegionKind, OcrText, Bounds), WindowCaptureTextIndex (FTS), plus separate SemanticTextStore.sidb / SemanticImageStore.sidb using DiskANN ANN — i.e. both text AND image embeddings, region-granular. Capture is policy-gated (12 policies: battery saver, idle, game mode...), default retention 90 days / 75 GB. Sensitive-content filtering uses Purview libraries (passwords, credit cards) and skips private browsing.
— https://github.com/xaitax/TotalRecall and https://blogs.windows.com/windowsexperience/2024/09/27/update-on-recall-security-and-privacy-architecture/

### Rewind is dead: Meta acquired Limitless (Dec 5, 2025) and the Rewind Mac app disabled all capture on Dec 19, 2025 — leaving a vacuum for a private, Mac-native replacement
Limitless (ex-Rewind AI) was acquired by Meta; the final update disables all screen/audio capture, and Pendant sales stopped. Ex-users explicitly cite the betrayal of the privacy-first promise (hardware pivot, cloud dependency, Meta) and are shopping for local-first alternatives; screenpipe and Fenn are aggressively marketing into this gap. ScreenMemory's 100%-local + encrypted-at-rest positioning lands exactly on the abandoned demand.
— https://9to5mac.com/2025/12/05/rewind-limitless-meta-acquisition/

### retrace (open-source Swift 'Rewind but better') shows what a ScreenMemory sibling chose: FTS5 keyword + app/date filters first, encryption and semantic search deferred
CGWindowListCapture every 2s (configurable 1-60s), frame dedup, Apple Vision OCR (200-500 ms/frame on Apple Silicon), HEVC via VideoToolbox (currently 50-70 GB/month vs Rewind's ~15 GB — storage efficiency is its acknowledged top gap). Search is SQLite FTS5 with query parsing, ranking, snippets, app/date/exclusion filters, <100 ms latency; no semantic search; AES-256-GCM via CryptoKit listed as 'in progress'. Also: rem (jasonjmcghee, 2.5k stars, semi-dormant since May 2024) never shipped embedding search — issue #17 stalled on sqlite-vss RAM/insert-speed concerns.
— https://github.com/haseab/retrace and https://github.com/jasonjmcghee/rem/issues/17

### Dayflow (6.1k stars, MIT, Swift, macOS) found a killer feature with NO search at all: 1fps capture -> VLM summarization every 15 min into a timeline of 'activity cards' + daily standup grid
It records screen chunks at 1fps and ships them to a user-chosen model (Ollama/LM Studio local, or Gemini/ChatGPT/Claude) to generate 15-minute activity summaries, rendered as a chronological journal with weekly analytics and a chat interface. Recordings auto-purge under a storage cap. Its traction proves the 'automatic work journal / daily recap' framing resonates more than 'search everything' — and it's achievable on-device with FoundationModels' 4k context (one 15-min window at a time).
— https://github.com/JerryZLiu/Dayflow

### User complaints cluster on four things: resource usage, transcript/search quality, recording-others consent, and pause controls — not on missing AI features
screenpipe HN thread: 'my CPU was hitting 700% and my fans started blasting on a MacBook M3 Max'; 'Transcription was terrible, even though whisper-large on its own works well'; 'recording both sides of a Zoom call without notice or consent'; 'I can't find anything in the docs about how to pause data collection'. A 2026 screenpipe review tells buyers to test exactly: CPU/battery impact, storage growth, search quality, accidental captures. Rewind's praised trait was the opposite: imperceptible overhead via Apple Silicon hardware accelerators (though Chen showed encoding was actually software libx264 and deferred on battery).
— https://news.ycombinator.com/item?id=41695840

### Windrecorder's economics trick: screenshot every 3s, index only on content/text change, then batch-convert screenshots into 15-min videos and do OCR/embedding work during idle time
Two modes (screenshot mode and 15-min video segments); pluggable OCR backends (Windows native, RapidOCR, WeChat OCR, Tesseract); all heavy lifting (compression, cleanup, OCR, image-embedding) is deferred to idle periods. It also ships activity statistics as a headline feature. Pattern worth copying: decouple capture (cheap, immediate) from indexing (deferred, batched) so the always-on path never competes with the user.
— https://github.com/yuka-friends/Windrecorder

### Across every system surveyed, the highest-value retrieval signal is structured metadata (app name, window title, browser URL, timestamp), captured via Accessibility APIs — ScreenMemory currently indexes none of it
Rewind's `segment` table (app/window + URL), screenpipe's accessibility_text + capture_trigger, Recall's WindowCapture (WindowTitle, ActivationUri, DwellTime) + App/Web/File relation tables all treat metadata as first-class filterable columns. This is what powers app filters, URL re-opening (Recall's Click-to-Do 'reopen source'), and grounded answers ('seen in Safari on linear.app at 14:32'). It is also nearly free to capture on macOS (NSWorkspace.frontmostApplication, AX window title, AppleScript/AX for browser URL).
— https://kevinchen.co/blog/rewind-ai-app-teardown/ and https://github.com/xaitax/TotalRecall

**Recommendations:**

- 1. Fix granularity before swapping models: stop embedding one averaged/truncated whole-screen vector (the proven-bad OpenRecall pattern, and your 128-token bug makes it worse). Use Vision's per-observation bounding boxes to group OCR lines into spatial blocks (title bar, main content, sidebar), embed each block (~100-300 chars) as its own row with frame_id + bounds — multiple vectors per frame, max-pool per frame at query time. This mirrors Microsoft Recall's ScreenRegion(OcrText, Bounds) design, the only region-granular system in the field.
- 2. Add SQLite FTS5 and make retrieval hybrid: every shipped system that users actually searched with (Rewind FTS4+porter, screenpipe FTS5, retrace FTS5, Recall WindowCaptureTextIndex) leans on keyword FTS; none relies on semantic-only, which is the weakest known configuration. Use FTS5 with unicode61 (+ trigram tokenizer for FR partial matches), fuse FTS rank and cosine with simple RRF. Note FTS5 stores text in the index — to keep your no-plaintext-on-disk guarantee, either accept indexed-token exposure with FileVault, or put the FTS index in an encrypted container (SQLCipher/SEE-style), as Recall does inside its enclave.
- 3. Capture app name, window title, and browser URL per frame via Accessibility/NSWorkspace and store them as columns (Rewind's `segment`, Recall's `WindowCapture`). Use them as (a) pre-filters ('in Slack last Tuesday'), (b) boost terms in hybrid ranking, and (c) grounding in the FoundationModels prompt ('Source: Safari — linear.app — 2026-06-10 14:32'). This single change does more for answer groundedness than any embedding upgrade, and enables the app-filter UI users expect.
- 4. Ship a Dayflow-style daily recap as the killer feature: batch each 15-min window's deduped OCR+metadata through FoundationModels (fits the 4k context one window at a time), store the summary cards, and render a timeline/day journal. Dayflow's 6.1k stars with NO search proves the 'what did I actually do today' journal outsells raw Q&A; with Rewind dead since Dec 2025, a private Mac-native recap+search combo has no incumbent.
- 5. Move toward event-driven capture with accessibility-tree text (screenpipe's model: capture on app switch/click/typing-pause/clipboard, AX tree first, Vision OCR fallback) and defer all indexing/embedding to idle time (Windrecorder's pattern). This addresses the #1 user complaint category in the wild — CPU/battery/fans — and AX text is cleaner input for embeddings than whole-screen OCR soup.
- 6. For grounded RAG within 4k context: retrieve top ~5 block-level chunks, collapse near-duplicate frames (same app+title+high text overlap) at query time so the context isn't 5 copies of one screen, and require the model to cite frame timestamps it used; add visible pause/per-app exclusion controls (a recurring complaint against screenpipe and a Recall table-stakes feature: private-browsing and sensitive-content exclusion by default).

## retrieval-sota

### OCR semantic noise alone causes up to ~50% retrieval/RAG performance decline, and BM25 degrades FASTER than dense retrieval as OCR noise increases
The OHRBench study (OCR Hinders RAG) shows even the best OCR pipelines lose 1.9 EM@1 / 2.93 F1@1, and semantic noise (misrecognized characters, typos) causes near-50% decline for most retriever+LLM combos. BM25 beats dense (BGE-M3) on clean text but falls below it as perturbation intensifies — directly relevant to ScreenMemory: keep the dense leg as the noise-robust backbone and treat lexical as a complement, not a replacement. Mitigation: prioritize OCR semantic accuracy (Vision 'accurate' path) over formatting.
— https://arxiv.org/html/2412.02592v1

### The single biggest lever for ScreenMemory is multi-vector-per-frame chunking: the seq_len=128 truncation means ~all but the first 500 chars of every screen are invisible to retrieval
Standard practice for documents longer than the encoder window is to split into 200-400 token chunks with 10-20% overlap, embed each, and score a document as max(chunk scores). For screen OCR, layout-aware blocks beat fixed-size: Apple Vision returns per-line bounding boxes (RecognizedTextObservation), and grouping lines into spatial clusters (X/Y proximity) yields coherent blocks (sidebar vs main content vs chat pane) instead of one flat string mixing UI chrome with content. The new RecognizeDocumentsRequest (WWDC 25) returns structured layout (paragraphs, tables, lists) natively, making manual grouping largely obsolete on macOS 26.
— https://weaviate.io/blog/chunking-strategies-for-rag and https://medium.com/@surajkumbhar904/ios-26-apple-adds-native-table-detection-to-vision-framework-142558ab086a

### Hybrid BM25 + dense fused with Reciprocal Rank Fusion (RRF, k=60) is the consensus fix for exactly ScreenMemory's failure mode: exact tokens (names, error codes, URLs, French proper nouns) that 512-d multilingual embeddings miss
RRF(d) = sum_i 1/(60 + rank_i(d)) over each retriever's ranked list — no score normalization needed (BM25 scores and cosine live on different scales), unsupervised, ~3 lines of code, consistently beats either retriever alone and other fusion methods on NDCG. Typical pipeline: BM25 top-50 + dense top-50 -> RRF -> top-k. At <10k rows the lexical index cost is trivial.
— https://atalupadhyay.wordpress.com/2026/06/10/building-a-production-ready-hybrid-retrieval-system-from-scratch-bm25-dense-embeddings-rrf-re-ranking/

### The encryption-at-rest constraint is solvable with an in-memory SQLite FTS5 database built at query/session start: FTS5 ships a native bm25() ranking function and works in ':memory:' databases
Pattern: decrypt rows (acceptable at <10k), insert into an in-memory FTS5 virtual table (use unicode61 tokenizer with remove_diacritics for FR+EN), query with MATCH ... ORDER BY bm25(). Nothing plaintext touches disk. Screenpipe (the leading open-source screen-memory project) uses exactly FTS5 virtual tables for its text search over screen captures. Build once per app session (not per query) and incrementally append new frames; rebuild cost for 10k short rows is sub-second.
— https://www.hisqlboy.com/blog/sqlite-full-text-search-fts5 and https://docs.screenpi.pe/architecture

### A simple recency prior — score = alpha*cos(q,d) + (1-alpha)*0.5^(age_days/h), alpha=0.7, half-life h=14 days — takes 'latest@10' accuracy from 0.00 to 1.00 on timestamped corpora, with no retraining
arXiv 2509.19376 evaluates this as a post-retrieval re-scoring layer: cosine-only baselines completely fail at 'most recent X' questions (Latest-Set@10 = 0.00 on the CERT logon dataset) while the fused score reaches 1.00. Sensitivity analysis: works for alpha <= 0.7, degrades at alpha >= 0.9. For ScreenMemory, screen data is intrinsically temporal, so this is near-free and should only apply when the query is not explicitly date-scoped (hard filters win then).
— https://arxiv.org/html/2509.19376

### Query date understanding ('yesterday', 'mardi dernier') should be a hard SQL time-range filter before vector search, and can be done offline in Swift via NSDataDetector or SoulverCore DateParsing
NSDataDetector (.date checking type) resolves many absolute and some relative expressions offline and is locale-aware (handles French), but misses compound phrases ('3 weeks from tomorrow'); soulverteam/DateParsing (SoulverCore, free for this use) handles richer natural-language dates in a Swift package. Alternative that fits the existing stack: ask the on-device FoundationModels model to emit a structured {date_from, date_to, cleaned_query} via guided generation before retrieval — standard 'temporal routing' practice in time-aware RAG. Range filter first, then dense+BM25 within the window.
— https://github.com/soulverteam/DateParsing and https://nshipster.com/nsdatadetector/

### For on-device reranking, mxbai-rerank-base-v2 (0.5B, Qwen2.5-based, 100+ languages incl. French, Apache-2.0, BEIR 55.57) is the best multilingual quality/size tradeoff; mxbai-rerank-xsmall-v1 (70.8M) is the smallest viable but English-leaning and weak on nuanced text
Cross-encoder reranking of the fused top-30..50 is the standard precision stage after RRF. Options by size: mxbai-rerank-xsmall-v1 (70.8M params, fast, but 'lacks capacity on longer or more nuanced texts'), jina-reranker-v2-base-multilingual (278M, strong FR support but CC-BY-NC license), mxbai-rerank-base-v2 (0.5B, Apache-2.0, BEIR 55.57 — beats much larger models, 8k context). A 0.5B Qwen-style decoder is convertible to CoreML fp16 (~1GB) and only runs on ~30 pairs per query, so latency is bounded; given ScreenMemory's tiny corpus, reranking is where most of the relevance gain over raw cosine will come from.
— https://www.mixedbread.com/blog/mxbai-rerank-v2 and https://huggingface.co/mixedbread-ai/mxbai-rerank-xsmall-v1

### Screenpipe and Rewind converge on the same lessons: prefer accessibility-tree text over OCR when available, capture on change events rather than fixed fps, and pair every text hit with its frame for grounding
Screenpipe (Rust, open source) extracts text via macOS accessibility APIs with OCR as fallback — AX text is noise-free, which sidesteps the OCR-noise retrieval degradation entirely for native apps — and stores in SQLite with FTS5 plus embeddings for semantic search. Rewind's reported pipeline reranks dense hits (cosine threshold ~0.8 in LanceDB) before passing to a local Llama-3.1-8B for answers. Both treat time/app/window metadata as first-class filterable columns, enabling 'in Slack last Tuesday' style queries.
— https://docs.screenpi.pe/architecture and https://ucstrategies.com/news/rewind-ai-mac-memory-search-tool-specs-privacy-pricing-2026/

### Near-duplicate collapsing at query time: capture-time avg-hash is not enough — group hits by (app/window, time bucket) and keep the best-scoring chunk per group, or apply MMR, before filling the small 4k context
Screen streams produce dozens of near-identical frames of the same document spread across minutes; cosine top-k then returns k copies of one moment and starves the 4k-token FoundationModels context. Practical fix used in production hybrid pipelines: after RRF/rerank, cluster results whose text similarity exceeds ~0.9 or that share window-title + a 2-5 min bucket, keep one representative each (with its timestamp), optionally diversify with maximal-marginal-relevance. This both grounds answers (distinct moments, citable timestamps) and frees context budget for actually distinct evidence.
— https://ashutoshkumars1ngh.medium.com/hybrid-search-done-right-fixing-rag-retrieval-failures-using-bm25-hnsw-reciprocal-rank-fusion-a73596652d22

### Evaluation: a 100-500 item golden set of query -> gold-frame/chunk pairs measuring Recall@k and MRR is the prerequisite for tuning everything above; synthetic queries can be generated locally from your own captured frames
Standard practice: label 100-500 (query, gold context, gold answer) tuples, target Recall@20 >= 0.8 and track MRR (how early the first correct frame appears); version the set so runs are comparable. For ScreenMemory, generate synthetic questions per frame with the on-device FoundationModels model ('what question would this screen answer?'), keep the frame id as ground truth — Ragas-style synthetic generation without any cloud call. Include temporal queries ('yesterday', 'last week') and FR+EN mix, since those are the failure modes being fixed.
— https://labelyourdata.com/articles/llm-fine-tuning/rag-evaluation

**Recommendations:**

- 1. Fix embedding coverage first: split each frame's OCR into layout-aware blocks using Vision bounding-box clustering (or RecognizeDocumentsRequest on macOS 26), ~200-400 tokens each with 10-20% overlap, embed every block (multiple 512-d vectors per frame), and score frames as max over their chunks. The seq_len-128 truncation means most screen text is currently unsearchable — no other change matters until this is fixed.
- 2. Add a lexical leg with RRF: at app-session start, decrypt rows into an in-memory SQLite FTS5 ':memory:' table (unicode61 tokenizer, remove_diacritics=2 for French), take BM25 top-50 + cosine top-50, fuse with RRF (k=60). This satisfies the no-plaintext-on-disk constraint and recovers exact-token queries (names, error codes, URLs) that the multilingual embedding misses.
- 3. Make retrieval time-aware: (a) pre-retrieval, extract date ranges from the query (NSDataDetector or a FoundationModels guided-generation call emitting {date_from, date_to, cleaned_query}) and apply as a hard SQL filter; (b) otherwise apply the recency prior alpha*cos + (1-alpha)*0.5^(age_days/h) with alpha=0.7, h~7-14 days as a re-scoring layer.
- 4. Collapse near-duplicates at query time before answering: group post-fusion hits by (window/app, 2-5 min time bucket) or text-similarity > 0.9, keep the best chunk per group with its timestamp, and build the FoundationModels prompt as a list of timestamped snippets with an instruction to cite times — this directly fixes ungrounded answers within the 4k context budget.
- 5. Add a CoreML cross-encoder reranker on the fused top-30: mxbai-rerank-base-v2 (0.5B, Apache-2.0, multilingual FR+EN, BEIR 55.57) converted to CoreML fp16; fall back to mxbai-rerank-xsmall-v1 (70M) if 0.5B latency/memory is unacceptable. With a <10k-row corpus this stage delivers most of the remaining precision.
- 6. Build the eval harness before tuning steps 2-5: ~150 (query -> gold frame) pairs, half hand-written from your own usage and half synthetically generated per-frame by FoundationModels, mixing FR/EN and temporal phrasings; track Recall@10 and MRR on every change. As a longer-term capture improvement, prefer macOS accessibility-tree text over OCR when available (Screenpipe's approach) to remove OCR noise at the source.

## embedding-models

### EmbeddingGemma-300m is still the top multilingual embedding model under 500M params on MTEB and is explicitly designed for on-device use
308M params, 768-d output with Matryoshka truncation to 512/256/128, 2048-token context, trained on 100+ languages (FR included), MMTEB v2 mean task score 61.15 at 768d. Google ships QAT checkpoints (Q4_0/Q8_0, int4 weights + int8 attention) and it runs in <200MB RAM quantized. Retrieval prompt prefixes are mandatory: 'task: search result | query: {q}' and 'title: none | text: {doc}'. License is the Gemma Terms of Use (commercial use allowed with use restrictions, not OSI-approved).
— https://huggingface.co/google/embeddinggemma-300m and https://huggingface.co/blog/embeddinggemma

### A working EmbeddingGemma CoreML/ANE port already exists with a Swift API - the conversion work is essentially done for you
The CoreML-LLM project (john-rocky) ships an EmbeddingGemma-300M bundle: 295MB, INT8, reported 99.80% ANE residency on A19 Pro, Matryoshka dims 768/512/256/128 exposed in the API. Swift usage is 'EmbeddingGemma.downloadAndLoad(...)' then 'eg.encode(text:, task: .retrievalQuery, dim:)'. Caveat: their default bundle is seq_len 128 (same truncation bug ScreenMemory has today), but the build tooling exposes sequence-length configuration, so re-converting at 512-1024 tokens is feasible.
— https://github.com/john-rocky/CoreML-LLM (docs/EMBEDDINGGEMMA.md)

### EmbeddingGemma activations do NOT support float16 - a real risk for ANE, which is natively fp16
The official model card states activations must be float32 or bfloat16; Gemma-family models are known to overflow fp16 (max 65504) in normalization/attention paths, producing NaN/-inf (documented in onnxruntime issue 26732 and vLLM issue 40290). On Core ML this means either INT8/QAT conversion (what CoreML-LLM did) or compute_precision=FLOAT32 islands for the offending ops, which can push parts of the graph off the ANE. Budget validation time: compare CoreML embeddings vs sentence-transformers fp32 cosine > 0.99 before shipping.
— https://huggingface.co/google/embeddinggemma-300m and https://github.com/microsoft/onnxruntime/issues/26732

### swift-transformers (v1.3.3, actively maintained) already supports both tokenizers you would need - you can delete the custom WordPiece constraint from the decision
Sources/Tokenizers/Tokenizer.swift registers 'XLMRobertaTokenizer' -> UnigramTokenizer.self (covers the whole E5/BGE/Arctic/Granite XLM-R SentencePiece-Unigram family) and 'GemmaTokenizer' -> BPETokenizer.self (covers EmbeddingGemma's 262k-vocab tokenizer via the Hub tokenizer.json). Pure Swift, loads via AutoTokenizer.from(pretrained:), no Python at runtime - same constraint profile as your current WordPiece implementation.
— https://github.com/huggingface/swift-transformers/blob/main/Sources/Tokenizers/Tokenizer.swift (lines 166, 175-176)

### multilingual-e5-small is the cheapest serious upgrade: retrieval-trained (unlike distiluse), MIT-licensed, and trivially ANE-convertible
Verified via HF API: 117.65M F32 params, MIT license, 8.3M downloads. 384-d, 512-token max, based on Multilingual-MiniLM-L12-H384 - a vanilla BERT-style encoder, exactly the architecture Apple's own 'Deploying Transformers on the Apple Neural Engine' work (ane_transformers) targets, so coremltools conversion at fp16 is routine with no overflow drama. Requires 'query: ' / 'passage: ' prefixes (asymmetric). distiluse-base-multilingual-cased-v2 was trained for sentence similarity, not retrieval, which is a large part of why ScreenMemory's relevance is 'terrible'; the E5 family was contrastively pretrained on ~1B text pairs specifically for retrieval. multilingual-e5-base (278M, 768d) is the same tokenizer/arch if more quality is wanted.
— https://huggingface.co/intfloat/multilingual-e5-small and https://machinelearning.apple.com/research/neural-engine-transformers

### snowflake-arctic-embed-m-v2.0 is the best Apache-2.0 multilingual retriever near 300M, but its GTE base makes CoreML conversion harder
305M total params (113M non-embedding), 768-d with MRL truncation to 256, 8192-token context via RoPE, Apache 2.0. It is built on gte-multilingual-base and beats it across MTEB-R (0.554), CLEF (0.534), MIRACL (0.592). The catch for ANE: the GTE architecture uses custom modeling code (RoPE + unpadded/xformers-style attention) rather than vanilla BERT, so there is no known CoreML/ANE port and conversion requires graph surgery - unlike the E5/XLM-R family.
— https://arxiv.org/abs/2412.04506 and https://huggingface.co/Snowflake/snowflake-arctic-embed-m-v2.0

### jina-embeddings-v3 is license-disqualified and bge-m3 / Qwen3-Embedding-0.6B are over the always-on power budget
jina-embeddings-v3 (572M, task-LoRA adapters) moved from Apache 2.0 to CC-BY-NC 4.0 - unusable in an open-source app others may use commercially. bge-m3 is MIT but 568M params, 1024-d, XLM-R-large based - roughly 5x the compute of e5-small per frame at 1fps always-on. Qwen3-Embedding-0.6B tops small-model multilingual MTEB (64.33), Apache 2.0, MRL to 1024-d, 32k context, but it is a 595M decoder-style model with instruction prompts - heavy for continuous background embedding, and no ANE conversion reports exist.
— https://huggingface.co/jinaai/jina-embeddings-v3, https://huggingface.co/Qwen/Qwen3-Embedding-0.6B, https://qwenlm.github.io/blog/qwen3-embedding/

### granite-embedding-278m-multilingual is a solid Apache-2.0 middle option on the same easy XLM-R conversion path as E5
IBM's granite-embedding multilingual models (107m: 384-d, 278m: 768-d, both 512-token, Apache 2.0, no prompt prefixes required) are XLM-RoBERTa-based, so they share the exact tokenizer (UnigramTokenizer in swift-transformers) and the trivially ANE-convertible BERT-style graph as multilingual-e5. Useful if the e5 'query:/passage:' prefix discipline is a problem, or if Apache 2.0 licensing purity matters more than the last few MTEB points; retrieval quality is broadly comparable to e5-base, below EmbeddingGemma.
— https://arxiv.org/abs/2508.21085 (Granite Embedding R2 paper) and https://huggingface.co/ibm-granite/granite-embedding-278m-multilingual

### Hugging Face's 'ember' project converts arbitrary sentence-transformers models to Core ML with GPU/ANE acceleration - a reusable conversion recipe
github.com/huggingface/ember auto-downloads any sentence-transformers model and converts it to Core ML ('GPU and ANE accelerated'), serving embeddings locally. It is early-stage (16 commits, no releases) so treat it as a conversion reference implementation rather than a dependency, but it demonstrates the standard coremltools path for the MiniLM/E5-class models and includes benchmark_results.txt for measured numbers.
— https://github.com/huggingface/ember

### Whatever the model, the seq_len-128 whole-screen embedding is the dominant relevance killer - chunking matters more than the model swap
Embedding ~first 500 chars of an entire screen's OCR into one 512-d vector destroys retrieval no matter the encoder: a screen mixes app chrome, menus, and content, and 128 tokens covers a fraction of it. All top candidates handle 512-2048+ tokens, and EmbeddingGemma/arctic support Matryoshka so multiple chunk vectors per frame at 256-d cost the same storage as today's single 512-d vector. The EmbeddingGemma document prefix even has a slot for structure: 'title: {app/window title} | text: {chunk}' - a natural fit for per-window OCR chunks.
— https://huggingface.co/google/embeddinggemma-300m (prompt format) and https://developers.googleblog.com/en/introducing-embeddinggemma/

**Recommendations:**

- 1. Quick win (days, biggest relevance-per-effort): swap distiluse for multilingual-e5-small (MIT, 118M, 384-d, 512 tokens). It is retrieval-trained where distiluse is not, converts to fp16 Core ML/ANE with stock coremltools (vanilla BERT graph, Apple's ane_transformers pattern applies), and swift-transformers' XLMRobertaTokenizer->UnigramTokenizer replaces your custom WordPiece. Implement the asymmetric 'query: '/'passage: ' prefixes correctly - skipping them measurably degrades E5.
- 2. Ship the fix that beats any model swap: stop embedding one truncated whole-screen vector. Chunk OCR output (per Vision text-region cluster or ~200-token windows, prefixed with app/window title), store several vectors per frame, retrieve with max-over-chunks cosine. Do this in the same PR as the e5 swap so you can A/B the two changes.
- 3. Target state (best quality under 500M): EmbeddingGemma-300m via the CoreML-LLM port (INT8, 99.8% ANE, Matryoshka 768/512/256/128, Swift API with .retrievalQuery task prefixes). Re-convert at seq_len 512-1024 instead of their 128 default, and validate against sentence-transformers fp32 outputs because Gemma activations overflow fp16 - use INT8/QAT, never naive fp16 conversion. Accept the Gemma license terms in your README.
- 4. Use Matryoshka 256-d for the stored vectors (EmbeddingGemma or arctic-m-v2.0): with multiple chunks per frame at 1fps, 256-d keeps your brute-force cosine scan and SQLite size in check at negligible quality loss; keep 768-d only if you later add a rerank stage.
- 5. Skip: jina-embeddings-v3 (CC-BY-NC license, task-LoRA complexity), bge-m3 and Qwen3-Embedding-0.6B (568-595M params, too heavy for an always-on 1fps ANE budget, no ANE ports), gte-multilingual-base/arctic-m-v2.0 as first choice (custom RoPE/unpadded-attention code means real CoreML conversion work - only revisit if you need 8k-token context and strict Apache 2.0).
- 6. Fallback if EmbeddingGemma's license or fp16 quirks block you: granite-embedding-278m-multilingual (Apache 2.0, 768-d, no prefixes, same XLM-R tokenizer and easy-conversion BERT graph as E5) as the quality step between e5-small and EmbeddingGemma.

## memory-systems

### RAPTOR-style hierarchical summarization is the single best-fit technique — VERDICT: HIGHLY APPLICABLE with a time-based tree instead of clustering
RAPTOR (ICLR 2024) recursively embeds, clusters (UMAP + Gaussian Mixture soft clustering, BIC for cluster count) and summarizes chunks bottom-up; summaries average 131 tokens with 72% compression (6.7 children per parent), generated by gpt-3.5-turbo — well within a ~3B on-device model's ability. Crucially, 'collapsed tree' retrieval (flatten all levels, cosine top-k under a 2000-token budget, ~20 nodes) beats tree traversal, which maps directly onto ScreenMemory's existing brute-force cosine search: just add hourly/daily/weekly summary rows to the same vector table. For a screen timeline you can replace GMM clustering with natural time buckets (screens -> hour -> day -> week), making indexing trivial. The 2000-token evidence budget fits Apple's 4096-token combined limit.
— https://arxiv.org/abs/2401.18059 (HTML: https://arxiv.org/html/2401.18059v1)

### Mem0's extract-then-consolidate pipeline (ADD/UPDATE/DELETE/NOOP) — VERDICT: APPLICABLE IN BATCHED FORM ONLY
Mem0 runs two LLM phases: an Extractor produces candidate facts from new content + a running summary, then an Update phase retrieves top-s similar existing memories and has the LLM pick one of four ops (ADD new, UPDATE/augment, DELETE contradicted, NOOP). It reports >90% token savings and 91% lower p95 latency vs full-context, +26% LLM-as-judge over OpenAI memory; the graph variant Mem0-g adds only ~2%. Per-fact LLM calls are too expensive at 1fps screen capture, but running extraction+consolidation in idle-time batches over deduped screen sessions is feasible — this is exactly the 'memory consolidation / merging near-duplicates' mechanism ScreenMemory lacks beyond avg-hash.
— https://arxiv.org/abs/2504.19413

### Letta's sleep-time compute pattern — VERDICT: ADOPT THE PATTERN (not the framework); perfect fit for an always-on Mac agent
Letta splits work between a primary agent (answers queries, read-only memory) and a sleep-time agent that runs during idle periods with exclusive memory-editing tools: it consolidates archival items, rewrites messy memory blocks, and summarizes recent activity into stable notes, in an 'anytime' fashion so the primary agent never blocks on it. It is model-agnostic. An always-on screen recorder has abundant idle windows (screen locked, overnight, on AC power) where the FoundationModels session is otherwise unused — ideal scheduling slots for summary generation, fact consolidation, and KG updates without impacting capture or query latency.
— https://www.letta.com/blog/sleep-time-compute

### Zep/Graphiti temporal KG — VERDICT: CONCEPTS YES, FRAMEWORK NO; and Apple's guided generation neutralizes its main small-model objection
Graphiti uses the LLM only at ingestion (entity/edge extraction + dedup); query time is LLM-free hybrid search (embedding cosine + BM25 + graph traversal, sub-second). Its bi-temporal model gives facts validity windows — contradicted facts are invalidated, not deleted, enabling 'what was true when' queries (Zep: 94.8% DMR, up to +18.5% on LongMemEval with 90% lower latency). It requires Neo4j/FalkorDB (non-starter on-device) and explicitly warns 'very small models frequently emit JSON that doesn't match the requested schema'. But Apple's @Generable constrained decoding gives 'strong guarantees that the model's output conforms to the expected format', eliminating exactly that failure mode — so a lightweight SQLite re-implementation (entities + edges tables with valid_from/valid_to) is realistic.
— https://arxiv.org/abs/2501.13956 and https://github.com/getzep/graphiti

### HippoRAG: LLM-free query-time graph retrieval via Personalized PageRank — VERDICT: APPLICABLE, the cheapest way to get multi-hop retrieval on-device
HippoRAG (NeurIPS 2024) builds a schemaless KG offline with an LLM, then at query time extracts query concepts and runs Personalized PageRank seeded on those entities to rank passages — no LLM call in the retrieval loop, +20% on multi-hop QA over standard RAG. PPR over a small personal entity graph stored in SQLite is a few hundred lines of Swift and runs in milliseconds, fitting the 100% offline constraint. This directly attacks 'retrieval relevance is terrible': queries like 'that Stripe invoice I saw last week' resolve through entity links rather than fragile whole-screen embeddings.
— https://arxiv.org/abs/2405.14831

### Full GraphRAG is out, but its cheap local variants (FastGraphRAG-style NLP extraction, LazyGraphRAG deferral) — VERDICT: APPLICABLE
Microsoft GraphRAG's index-time community summarization is too LLM-heavy, and even GraphRAG-local-ollama guidance notes small models extract much less accurately than GPT-4-class models. Two adaptations fit on-device: (1) FastGraphRAG-style non-LLM extraction — entities as noun phrases, relationships as co-occurrence — which on macOS means Apple's NaturalLanguage NLTagger NER at near-zero cost on every OCR'd screen; (2) LazyGraphRAG's principle of skipping summarization at index time (~99% cheaper indexing) and summarizing on demand at query time, reserving the scarce 4k-token model budget for queries actually asked.
— https://blog.premai.io/graphrag-implementation-guide-entity-extraction-query-routing-when-it-beats-vector-rag-2026/ and https://microsoft.github.io/graphrag/

### MemoryBank's Ebbinghaus forgetting curve — VERDICT: APPLICABLE AND TRIVIAL; pairs with hierarchical summaries as a lossy retention policy
MemoryBank (AAAI 2024) modulates a per-memory strength score with exponential decay over time, reinforced on each recall — no LLM needed, just arithmetic at retrieval time. It also independently invented the hierarchy ScreenMemory needs: conversation logs -> daily event summaries -> global user profile. For screen data this becomes the retention policy: raw frames + full OCR text decay and get deleted after N days unless recalled, while hourly/daily summaries persist — bounding the encrypted SQLite store's growth (a real problem at 1fps) while keeping long-term recall.
— https://arxiv.org/abs/2305.10250

### A-MEM Zettelkasten-style agentic memory — VERDICT: NOT PRACTICAL on a 4k-context on-device model
A-MEM (NeurIPS 2025) generates a structured LLM note (context description, keywords, tags) for every new memory, then runs further LLM passes to link it to historical memories and retroactively evolve their attributes. At screen-capture data rates this means continuous multi-call LLM chains per memory; its gains were demonstrated on conversational memory with frontier/served models. The keyword/tag idea survives in cheaper form (NLTagger + one guided-generation call per session, not per memory), but the dynamic link-evolution loop should be skipped.
— https://arxiv.org/abs/2502.12110

### LongMemEval's indexing ablations give the highest-leverage retrieval fixes: fact-granularity values, fact-augmented keys, time-aware query expansion
LongMemEval (ICLR 2025) decomposes memory design into indexing/retrieval/reading and measures: storing at round/fact granularity beats whole-session storage; concatenating extracted facts as extra index keys adds ~4% recall@k and ~5% end-to-end QA; parsing the time range from the query and restricting retrieval to matching time blocks adds 7-11% on temporal questions. This is a direct indictment of ScreenMemory's current design — one 512-d embedding of a whole screen, truncated to ~500 chars by the seq_len-128 bug, is the worst point on the granularity curve. Chunking OCR text into multiple embeddings per screen plus timestamp-filtered search are pure-engineering wins requiring no LLM.
— https://arxiv.org/abs/2410.10813

### Episodic-memory theory says context binding (when/where/why) is the defining property — screen metadata must be first-class, and screenpipe shows how
The position paper 'Episodic Memory is the Missing Piece for Long-Term LLM Agents' defines five required properties: long-term storage, explicit reasoning over memory, single-shot learning, instance-specific memories, and contextual relations binding when/where/why to content. For a screen history, 'where' = app name, window title, browser URL — which screenpipe (the open-source Rewind successor, MIT) captures via the macOS accessibility tree first with OCR only as fallback, stores in SQLite FTS5, and exploits in filtered search; it also ships a 'day-recap' pipe (accomplishments, key moments, unfinished work) as its headline on-device feature. ScreenMemory currently embeds anonymous raw text blobs with none of this episodic context.
— https://arxiv.org/abs/2502.06975 and https://github.com/screenpipe/screenpipe

### The Apple on-device model is explicitly built for the exact operations these architectures need — summarization and entity extraction — but nothing more
The on-device foundation model is ~3B parameters, 2-bit QAT-quantized with 37.5% KV-cache sharing, supports 15 languages (150k-token vocabulary, French covered), and Apple's recommended uses are literally 'summarization, entity extraction, text understanding, refinement' — while warning it is 'not designed to be a chatbot for general world knowledge'. @Generable guided generation uses constrained + speculative decoding with format guarantees, and the Tool protocol handles parallel/serial tool-call graphs. iOS/macOS 26.4 added SystemLanguageModel.contextSize and tokenCount(for:) for budgeting the 4096-token combined input+output limit. Implication: use it as a batch summarizer/extractor feeding deterministic retrieval, not as the reasoning engine.
— https://machinelearning.apple.com/research/apple-foundation-models-2025-updates and https://www.infoq.com/news/2026/03/apple-foundation-models-context/

### Across 2025-2026 agent-memory surveys, consolidation is flagged as 'the most important and least implemented' mechanism — and the dedup risk is documented
Recent surveys (e.g. 'Memory for Autonomous LLM Agents', arXiv 2603.07670; 'Rethinking Memory Mechanisms of Foundation Agents', arXiv 2602.06052) converge on consolidation — turning raw episodic experience into reusable semantic knowledge — as the top open gap, and warn that consolidation without careful deduplication causes overwriting of still-relevant earlier episodes. For ScreenMemory this validates prioritizing the summarize+merge pipeline over more exotic techniques, and argues for non-lossy consolidation (keep provenance links from summaries back to raw frame rows, Graphiti-style invalidate-don't-delete) rather than destructive merging.
— https://arxiv.org/html/2603.07670v1 and https://arxiv.org/pdf/2602.06052

**Recommendations:**

- 1. Fix retrieval before adding any LLM machinery (LongMemEval evidence): chunk each screen's OCR text into ~100-200-token windows with multiple embeddings per frame (kills the seq_len-128 truncation bug as a side effect), add SQLite FTS5/BM25 hybrid search alongside cosine, attach app/window-title/URL/timestamp metadata to every chunk, and implement time-aware query filtering (parse 'last Tuesday' to a timestamp range before ranking). All deterministic Swift, no LLM, and it addresses the 'retrieval relevance is terrible' pain directly.
- 2. Build a time-bucketed RAPTOR tree as the consolidation + killer-feature layer: during idle/sleep-time windows (Letta pattern: screen locked, on AC), have FoundationModels generate ~130-token guided-generation summaries per hour from deduped screen text, then daily from hourlies, weekly from dailies; embed every summary into the same vector table and retrieve collapsed-tree style under a ~2000-token budget. The daily/weekly summaries double as an automatic 'day recap / what was I working on' feature (screenpipe's day-recap proves demand) — the missing killer feature beyond Q&A.
- 3. Add a lightweight personal entity graph in SQLite, not a graph DB: NLTagger NER on every frame for free entity candidates, one @Generable extraction call per idle batch to normalize entities and relations (constrained decoding guarantees valid schema even at 3B params), bi-temporal valid_from/valid_to columns Graphiti-style (invalidate, never delete), and HippoRAG-style Personalized PageRank seeded from query entities at search time — zero LLM calls in the retrieval loop, multi-hop capability for 'that doc Paul sent about the migration'.
- 4. Implement an Ebbinghaus retention policy (MemoryBank): per-memory strength score decaying exponentially, reinforced on recall; raw frames and full OCR text are purged after N days unless recalled, hourly summaries after months, daily/weekly summaries kept indefinitely. Bounds the encrypted store's growth at 1fps and is pure arithmetic.
- 5. Adopt Mem0's ADD/UPDATE/NOOP consolidation only for a small 'durable facts' tier (user's projects, recurring contacts, tools, preferences) updated once per idle cycle — retrieve top-s similar existing facts, let the on-device model pick the operation via a @Generable enum. Skip A-MEM-style per-memory note generation and link evolution entirely; it does not fit a 4k-context 3B model at screen-capture rates.
- 6. Switch primary text capture to the macOS accessibility tree with Vision OCR as fallback (screenpipe's approach): cleaner text than whole-screen OCR, structural context (button vs document body) for free, and better French text quality — improving every downstream stage (embeddings, summaries, entity extraction) at no model cost.

## apple-stack

### FoundationModels on-device model has a hard 4,096-token context window per LanguageModelSession, and EVERYTHING counts against it
Apple Technote TN3193 confirms 4,096 tokens max per session, with ~3-4 chars/token for Latin languages. Instructions, prompts, all responses, tool definitions, tool outputs, @Generable JSON schemas, @Guide annotations, and the full transcript all consume the budget. There is a public token counter (await model.tokenCount(for:)), a catchable LanguageModelSession.GenerationError.exceededContextWindowSize error, and a documented recovery pattern: rebuild a new session from a condensed Transcript (e.g. keep first+last entries) then prewarm(). A Foundation Models instrument in Xcode profiles token usage. Practical consequence for ScreenMemory: RAG context must fit in roughly 2.5-3k tokens, i.e. 3-5 small OCR chunks max, not whole screens.
— https://developer.apple.com/documentation/technotes/tn3193-managing-the-on-device-foundation-model-s-context-window

### TN3193 explicitly endorses the RAG architecture ScreenMemory uses, and prescribes the prompting style that makes the 3B model reliable
Apple's own guidance for large knowledge bases: chunk, embed (it names the NaturalLanguage framework for embeddings), store, retrieve top-k, pass only relevant chunks. Prompting rules for the small model: 1-3 paragraphs max, imperative single-task prompts ('List 3...'), bound output length via @Guide(maximumCount:) rather than maximumResponseTokens (which truncates mid-structure), 3-5 tools max per request, short property names in Generable types ('screen space ~ token usage'). The WWDC25 session 'Explore prompt design & safety' (session 248) adds: the model is for summarization/extraction/classification, not world knowledge or reasoning — extraction-only patterns are exactly its sweet spot.
— https://developer.apple.com/documentation/technotes/tn3193-managing-the-on-device-foundation-model-s-context-window and https://developer.apple.com/videos/play/wwdc2025/248/

### @Generable guided generation uses constrained decoding, so schema-valid structured output is guaranteed — this is THE reliability mechanism production apps use with the weak 3B model
The @Generable and @Guide macros (WWDC25 sessions 286 'Meet the Foundation Models framework' and 301 'Deep dive') make the runtime mask invalid tokens during sampling, so the model cannot produce malformed output; no JSON parsing or retries needed. Tool calling (Tool protocol with a nested @Generable Arguments struct) gets type-safe arguments the same way. streamResponse delivers PartiallyGenerated snapshots (progressively-filled optional properties) rather than raw text deltas, ideal for live UI. For grounded RAG answers, the pattern is a Generable struct like {answer: String, sourceChunkIDs: [Int], couldNotFind: Bool} so hallucination is reduced to choosing among provided IDs.
— https://developer.apple.com/videos/play/wwdc2025/301/ and https://developer.apple.com/documentation/FoundationModels

### Rate limits only bite background apps — critical for an always-on agent like ScreenMemory
LanguageModelSession.GenerationError.rateLimited fires when an app exceeds a system-defined rate limit while running in the background (foreground apps are effectively unlimited); also each session handles one request at a time (concurrent respond() calls throw). So a background daemon that continuously summarizes captured screens will hit rateLimited and must queue/back off; heavy LLM work (e.g. nightly digests) should run when the app is frontmost or be batched. Adapters exist too: Apple ships an adapter training toolkit (Python, rank-32 LoRA + optional draft model for speculative decoding) but adapters are tied to specific base-model versions and require entitlements — high maintenance, last-resort option.
— https://developer.apple.com/documentation/foundationmodels/languagemodelsession/generationerror/ratelimited(_:) and https://developer.apple.com/apple-intelligence/foundation-models-adapter/

### RecognizeDocumentsRequest (new in macOS 26 Vision) returns structured layout — paragraphs, tables, lists, detected data — and is the direct fix for ScreenMemory's flat OCR text
WWDC25 session 272 'Read documents using the Vision framework': RecognizeDocumentsRequest produces a hierarchical DocumentObservation (containers of text/tables/lists/barcodes), groups lines into paragraphs, recognizes 26 languages, and surfaces detected data (emails, phone numbers, URLs) — all on-device. Each element carries geometry, so you can emit layout-aware text blocks (window regions, sidebars, tables) instead of one raw text blob. For the classic API, VNRecognizeTextRequest with .accurate level supports French; set recognitionLanguages = ["fr-FR","en-US"] (priority order) plus automaticallyDetectsLanguage = true, and each VNRecognizedTextObservation gives a boundingBox CGRect usable for spatial chunking.
— https://developer.apple.com/videos/play/wwdc2025/272/ and https://developer.apple.com/documentation/vision/recognizedocumentsrequest

### NLContextualEmbedding (NaturalLanguage framework) is a built-in multilingual BERT-class embedding that could replace the custom distiluse CoreML model — 512-d Latin-script model covers FR+EN, 768-d reported on macOS
WWDC23 session 10042 'Explore Natural Language multilingual models': three transformer models grouped by script (Latin model = 20 languages incl. French and English), assets downloaded on demand (requestAssets), runs via CoreML on Apple silicon accelerators. It returns token-level vector sequences (NLContextualEmbeddingResult) — you mean-pool to get a sentence vector. Community benchmarks (buh/NaturalLanguageEmbeddings) report Precision@1 ~1.0 / MRR 0.83 on FAQ-style search, ~256-token sequence length (2x ScreenMemory's current 128), but high baseline similarity (0.60-0.89) between unrelated single words — chunking at 200-300 words with overlap recommended. It eliminates the self-maintained WordPiece tokenizer and CoreML conversion entirely; quality vs distiluse must be benchmarked, as it was designed as a fine-tuning feature extractor, not a contrastively-trained sentence encoder.
— https://developer.apple.com/videos/play/wwdc2023/10042/ and https://developer.apple.com/documentation/naturallanguage/nlcontextualembedding and https://github.com/buh/NaturalLanguageEmbeddings

### sqlite-vec works on macOS as a zero-dependency loadable extension doing SIMD brute-force search — and the right way to satisfy the encryption constraint is SQLCipher full-file encryption underneath it, not encrypted blobs
sqlite-vec (asg017) is pure C, runs anywhere SQLite runs, supports exact brute-force KNN with SIMD kernels, int8 and binary quantization, and Matryoshka slicing — no HNSW/ANN yet, but brute force at ScreenMemory's scale (tens of thousands to low millions of 512-d vectors) is fine. Vectors live in vec0 virtual-table shadow tables inside the same .db file, so it cannot search application-encrypted blobs — but compiled into SQLCipher (AES-256 page-level encryption, drop-in SQLite fork) the entire file including vector shadow tables is encrypted at rest while queries run on plaintext pages in memory. This also fixes ScreenMemory's current hole: vectors stored raw are invertible back to text (embedding inversion), so 'encrypted text, raw vectors' is not actually private.
— https://alexgarcia.xyz/blog/2024/sqlite-vec-stable-release/index.html and https://github.com/asg017/sqlite-vec and https://github.com/sqlcipher/sqlcipher

### Apple has published reference code for transformers and CLIP-style embedding models on the Neural Engine
apple/ml-ane-transformers (with the 'Deploying Transformers on the Apple Neural Engine' research post) is the canonical reference for restructuring a distilbert-class model for ANE-friendly execution (the exact family of ScreenMemory's distiluse model). apple/coreml-mobileclip on Hugging Face ships ready-made CoreML text+image encoders (text encoder runs ~102/108 ops on ANE) — relevant if ScreenMemory adds visual embedding of screenshots (search screenshots by what they LOOK like, not just OCR text). MobileCLIP2 (TMLR 2025) is the current best small CLIP from Apple.
— https://machinelearning.apple.com/research/neural-engine-transformers and https://huggingface.co/apple/coreml-mobileclip and https://github.com/apple/ml-mobileclip

### Next generation (announced WWDC 2026 / AFM 3) raises the on-device ceiling: a 20B sparse 'Core Advanced' model with image input that streams experts from flash
Apple ML Research 'Introducing the Third Generation of Apple's Foundation Models': AFM 3 Core stays 3B dense (45.6% human preference vs 23.3% for the 2025 model), AFM 3 Core Advanced is 20B sparse using Instruction-Following Pruning, with full weights in NAND flash and routed experts paged into DRAM on demand; expanded multilingual coverage explicitly includes French. Developer commentary notes image input and continued free access via the Foundation Models framework. This lands in the macOS-26-successor timeframe, so any architecture built on FoundationModels sessions/@Generable today gets a large free quality bump; don't over-invest in workarounds (e.g. adapters) for the current 3B's weaknesses.
— https://machinelearning.apple.com/research/introducing-third-generation-of-apple-foundation-models and https://ofox.ai/blog/apple-foundation-models-3-wwdc-2026-developer-read/

### Session mechanics that matter in practice: prewarm(), transcript-based session rebuild, GenerationOptions, and includeSchemaInPrompt
LanguageModelSession is stateful with a readable Transcript; you can construct a new session from an edited transcript (Apple's documented pattern for sliding-window memory). prewarm() loads the model ahead of first token to cut latency for interactive Q&A. GenerationOptions exposes temperature and greedy sampling — for extraction-only RAG answers use greedy/low temperature for determinism. When providing few-shot examples of the output format, respond(..., includeSchemaInPrompt: false) avoids paying schema tokens twice (WWDC25 deep-dive session 301). SystemLanguageModel.default.availability must be checked (Apple Intelligence off / model not downloaded / device unsupported are distinct unavailable reasons).
— https://developer.apple.com/documentation/foundationmodels/languagemodelsession and https://developer.apple.com/videos/play/wwdc2025/301/

**Recommendations:**

- 1. Fix retrieval at the chunking layer first: replace the single whole-screen embedding (truncated to ~500 chars by seq_len 128) with layout-aware blocks from RecognizeDocumentsRequest (paragraphs, tables, lists + bounding boxes, FR among 26 languages) and embed each block separately (multiple vectors per frame, frame_id + rect metadata). This one change attacks the root cause of terrible relevance; no model change needed yet.
- 2. Run a 1-day bake-off on a small FR/EN screen-text retrieval set: current distiluse (with chunking fix) vs built-in NLContextualEmbedding (Latin-script model, mean-pooled, ~256-token window, asset-managed by the OS). If NLContextualEmbedding is within a few points, adopt it and delete the CoreML model + WordPiece tokenizer — large maintenance win; if not, keep distiluse but raise seq_len and chunk to fit.
- 3. Restructure the answer stage around guided generation per TN3193: retrieve top 3-5 chunks (budget ~2.5k of the 4,096 tokens, verified with tokenCount(for:)), one fresh prewarmed session per question, greedy sampling, and a @Generable answer type {answer, sourceChunkIDs, couldNotFind} so the model can only cite provided chunks — this converts the weak 3B from a hallucinating chatbot into a constrained extractor and gives free grounded citations (click-through to the original screenshot moment).
- 4. Migrate storage to SQLCipher (full-file AES-256) + sqlite-vec compiled in: solves 'no plaintext index on disk' properly (current raw vectors are invertible to text, so today's at-rest story is broken), keeps SQL, and gets SIMD brute-force KNN with int8/binary quantization headroom to millions of frames without an ANN index.
- 5. Build the killer feature on tool calling + streaming: register a searchScreenMemory Tool (Generable arguments: query, time range, app filter) so the model does multi-hop retrieval itself ('what was that pricing page I saw after the call with X?'), and ship streamed daily/weekly timeline digests rendered from PartiallyGenerated snapshots. Schedule digest generation while the app is foreground or with backoff handling for GenerationError.rateLimited, since background LLM use is rate-limited.
- 6. Do NOT invest in custom LoRA adapters now: they are rank-32, version-locked to the base model and entitlement-gated, and AFM 3 (3B refresh + 20B sparse Core Advanced with image input, better French) arrives via the same FoundationModels API — keep the architecture session/@Generable-native and inherit the upgrade for free.

## Contradictions between researchers

1. **FTS5 vs the encryption constraint — three incompatible answers.** oss-landscape says "accept indexed-token exposure with FileVault or use SQLCipher"; retrieval-sota says rebuild an in-memory `:memory:` FTS5 at session start; apple-stack says migrate the whole store to SQLCipher + sqlite-vec. These are different architectures with different threat models and startup costs. A decision is needed, not a menu — and nobody costed the in-memory option (decrypt + reindex the full corpus on every launch; also still swappable to disk unless memory is locked).

2. **Daily-recap scheduling directly conflicts with rate limits.** value-features rec 2: "scheduled morning job (launchd)" running FoundationModels batches. apple-stack: background LLM use is rate-limited ("rate limits only bite background apps") and digests should run while foregrounded. The consensus killer feature as specced by one researcher is throttled by the platform constraint documented by another. Unresolved.

3. **Retention vs timeline scrubber.** memory-systems rec 4 purges raw frames/full OCR after N days (Ebbinghaus); value-features rec 3 builds the trust mechanic on opening "the actual screenshot at that moment" and rec 2/5 cite old sessions. Citations and timeline links will dangle into purged data. No reconciliation.

4. **Embedding model: three different endpoints.** e5-small → EmbeddingGemma (embedding-models), keep-distiluse-or-NLContextualEmbedding bake-off (apple-stack), "model swap is secondary" (retrieval-sota, memory-systems). Compatible in spirit (chunk first), but each swap forces re-embedding the whole historical corpus — a migration cost no one estimated, and doing the chunking fix + model swap "in the same PR" (embedding-models rec 2) contradicts the A/B discipline retrieval-sota's eval-first rec implies.

5. **Reranker power budget inconsistency.** embedding-models disqualifies Qwen3-Embedding-0.6B (~595M) as "over the always-on power budget", while retrieval-sota recommends mxbai-rerank-base-v2 (0.5B, also Qwen2.5-based) — defensible (query-time vs always-on duty cycle) but never argued, and its CoreML conversion difficulty is hand-waved while embedding-models flags much smaller GTE conversions as "real work".

## Suspicious claims

- **"<10k-row corpus" (retrieval-sota rec 5)** is inconsistent with always-on 1fps capture. Even with avg-hash dedup, multi-vector-per-frame chunking yields hundreds of thousands to millions of rows within months. This assumption silently underwrites brute-force cosine, in-memory FTS rebuild, and reranker latency claims. Nobody did the rows/day arithmetic.
- **BM25 tension:** retrieval-sota states BM25 degrades *faster* than dense under OCR noise, then every researcher's #2 rec is FTS5/BM25 hybrid over OCR text. RRF hedges this, but the lexical leg's value depends on the OCR cleanliness fix (AX tree) that is everywhere deferred to "longer term".
- **AFM 3 / "20B sparse Core Advanced, WWDC 2026"** is presented as a planning datum but is (at best) a days-old announcement; deferring LoRA and capacity decisions on it should be marked speculative, not factual.
- **"Raw vectors are invertible to text"** (apple-stack) — directionally right (Vec2Text-class attacks) but overstated as "broken"; inversion of 512-d pooled embeddings is approximate. Still worth acting on, just not with that certainty.

## Gaps worth flagging

1. **No capacity/latency model of the system.** Frames/day after dedup, chunks/frame, vector count at 6/12 months, brute-force scan latency, DB size, session-start decrypt time, and historical re-embedding cost — every major rec (chunking, hybrid, reranker, in-memory FTS) depends on these numbers and none were computed.
2. **No end-to-end answer-quality eval.** Only retrieval-sota proposes a golden set, and only for Recall@k/MRR. Nothing measures groundedness/hallucination of the final FoundationModels answer, FR vs EN splits, or ScreenMemory's *own* Vision OCR error rate (the cited 50%-degradation figure is generic; the local baseline is unmeasured).
3. **Capture-side privacy controls absent from the plan.** oss-landscape identifies consent/pause as a top-4 user complaint and Recall-style sensitive-content exclusion as table stakes, yet no researcher's roadmap includes pause, per-app/private-browsing exclusion, or secure-input-field detection — and value-features' encrypted JPEG keyframes materially expand the sensitive-data surface without a corresponding security analysis.
4. (Minor) **AX-tree vs RecognizeDocumentsRequest as primary text source** are presented as parallel fixes by different researchers; they have different coverage failure modes (Electron/canvas apps vs OCR noise) and nobody compared them or noted that switching text source invalidates the visual avg-hash dedup trigger.
